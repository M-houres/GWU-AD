# MVP 基线与项目报告

最后更新：2026-04-18

## 一、当前结论

项目现在按“学术文稿处理 MVP”收敛，不再按大而全平台推进。

MVP 只验证一件事：用户能稳定完成登录、充值通用点数、提交核心任务、获得结果；运营能在后台追踪用户、订单、任务、配置和算法状态。

基线之外的增长、裂变、代理、评审、答辩、独立日志页、独立权限页全部冻结或删除。后续如果要恢复，必须先重新更新本报告和 `gw工作日志.md`，不能直接补代码。

## 二、MVP 六条核心基线

### 1. 账户与短信登录基线

必须保留：

- 手机号验证码登录。
- 用户态与管理态会话。
- 微信登录与小程序登录配置能力。
- 登录风控参数。
- 新用户初始通用点数。

验收口径：

- 用户能登录并进入三类核心任务页。
- 管理员能登录后台。
- 登录失败、验证码错误、风控限制有明确提示。

### 2. 通用点数与支付基线

必须保留：

- 通用点数作为唯一对外账户口径。
- 6 档默认套餐：
  - 入门版：19 元，10,000 通用点数
  - 基础版：39 元，20,000 通用点数
  - 专业版：79 元，50,000 通用点数
  - 增强版：149 元，100,000 通用点数
  - 高级版：419 元，300,000 通用点数
  - 旗舰版：1199 元，1,000,000 通用点数
- 微信支付、支付宝支付、mock 支付联调能力。
- 订单创建、支付、到账、退款、幂等处理。
- 通用点数流水。

验收口径：

- 前台购买页只展示后端配置的套餐。
- 后台保存套餐不会丢失 `credits`。
- 充值后用户余额、订单、流水一致。
- 退款后订单状态和余额回滚可追踪。

### 3. 三类任务处理基线

只保留三类核心任务：

- `aigc_detect`：AIGC 检测。
- `dedup`：降重复率。
- `rewrite`：降 AIGC 率。

必须保留：

- 文件上传。
- 任务预处理。
- 字符数统计。
- 通用点数扣费。
- AIGC 每日免费额度。
- Celery / 本地队列兜底。
- 任务状态流转。
- 结果下载。
- 失败自动退款。

验收口径：

- 三类任务都能从提交走到状态可查。
- 扣费失败时提示“通用点数不足”。
- 处理失败时自动退回通用点数。
- 用户中心和后台任务页能看到一致结果。
- `dedup` 与 `rewrite` 的辅助报告改为可选，不再作为 MVP 主链必填项，也不在前台页面展示。
- 前台提交页、记录页、后台任务页都必须能看到任务状态、扣费点数、失败原因和退款状态。

### 4. 算法包与 LLM 基线

必须保留：

- 算法包上传、安装、激活、停用。
- 内置算法包引导。
- 当前三类任务的算法执行链。
- LLM 配置、开关、模型、API Key、超时、重试参数。
- LLM 与规则/算法链路的降级共存。

验收口径：

- 后台可以看到算法包状态。
- 每类核心任务有明确执行路径。
- LLM 关闭时主链不崩。
- LLM 异常时有日志或错误提示可排查。

### 5. 小程序基线

必须保留：

- 小程序登录配置。
- 小程序支付配置。
- 小程序 API 域名配置。
- 客户端来源 `source=miniprogram` 的传递。
- 小程序用户、订单、任务、流水来源追踪。

验收口径：

- 小程序来源登录后，用户来源不丢。
- 小程序来源订单和任务可在后台追踪。
- 小程序支付配置不被后台精简误删。

### 6. 后台运营基线

只保留 MVP 必需后台入口：

- 总览看板。
- 用户管理。
- 用户详情。
- 任务管理。
- 订单管理。
- 算法配置。
- 配置中心。

必须保留的配置：

- 登录 / 短信 / 微信 / 小程序登录。
- 支付。
- 通用点数套餐与任务计费。
- LLM。
- 小程序。
- 前台导航。
- 公告基础配置。

验收口径：

- 运营能查用户余额和流水。
- 运营能调整用户通用点数。
- 运营能查任务、下载结果。
- 运营能查订单并退款。
- 运营能配置套餐、支付、短信、LLM、小程序、算法。

## 五、模块化推进原则

### 1. 新功能增加原则

- 先判断属于哪个大模块，再决定放在哪个小模块。
- 不能因为“改起来方便”就继续往 `processing_engine.py`、`admin.py`、`tasks.py`、`worker_tasks.py` 这类巨型文件堆逻辑。
- 知网 / 维普 / 降AIGC率 / 降重复率 / AIGC检测等能力，优先进入“策略中心模块”和“处理编排模块”。

### 2. 问题修复原则

- 先定位问题属于哪个大模块。
- 再在该模块内定位具体小模块。
- 禁止在不相关模块打补丁式修复。

### 3. 结构重构原则

- 先收职责边界，再搬目录。
- 先消除重复规则实现，再做代码分拆。
- 先拆高风险巨型文件，再拆普通页面或普通接口文件。

### 4. 日志延续原则

- 后续所有模块调整、边界变更、结构重构、重要修复前后，必须继续写 `gw工作日志.md`。
- 日志要明确说明：
  - 改了哪个模块。
  - 为什么改。
  - 是否影响后续扩展或排错。
  - 后来的人 / 工具接手时应延续写日志，不得中断。

处理原则：

- 不再新增 runtime 引用。
- 不再新增页面、接口或测试。
- 迁移方案明确前，不直接删除历史表模型。

## 六、当前风险与边界

### 1. 历史模型暂未删除

原因：删除模型可能影响既有数据库、历史流水和迁移兼容。当前做法是切断运行入口，而不是直接拆库。

后续如果要彻底清库，需要单独做迁移方案：

- 确认线上是否存在 referral / promo 历史数据。
- 确认是否需要导出归档。
- 新增迁移删除表和枚举。
- 跑完整回归。

### 2. 前端 runtime 日志仍被进程占用

本轮已尝试删除本地 runtime 日志，其中前端两个日志文件被运行中的进程占用，暂时未删除：

- `frontend/runtime_stderr_local.log`
- `frontend/runtime_stdout_local.log`

这两个文件不属于源码基线，停掉本地前端进程后可删除。

### 3. 自定义金额充值仍是边界能力

主流程已经回到套餐购买，但后端仍保留自定义金额充值能力。MVP 下一步应决策：

- 如果坚持纯套餐制，就关闭 `amount_cny` 下单入口。
- 如果保留自定义充值，就必须在前台和后台明确隔离，不允许影响套餐真相源。
